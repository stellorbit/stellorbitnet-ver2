# 「軌道を描こう・改」誕生秘話 ～ 「SSG（静的サイトジェネレーター）」と出会い、格闘し、面白さを感じたお話 ～

本題の前に少し小咄を。

実はこのサイトは、もともと「はてなブログ」に開設していた **「軌道を描こう」** というブログの改良版の位置づけです。

今回はそれに出会い、戦い抜いた記録をお届けしたいと思います。

## Webサイト作るのがめんどいと前々から思っていた

序章の通り、これまではAWSだったりレンタルサーバー等でWordPressを使ってWebサイトを構築していました。

しかし、WordPressは便利なように見えて色々と厄介だったりします。**セキュリティ面**だとか**高速化**だとか。

### 実は安定・安全な運用のハードルが高いWordPress

WordPressは、本当に一見簡単そうに見えます。
AWSやGC・Azureなどで構築するにしても、先人が叩いてたコマンドをコピペすれば構築できます。
あとはテーマ入れて、テーマ設定の中でデザイン決めて・・・

しかし、それが罠。

WordPressの仕組み自体が **複雑かつ簡単（矛盾）** なので、色々不都合が生じたりします。

#### 1.WordPressで使われている「SQL」は格好の獲物

まず、SQL周りなんてのは、WordPressがどういう仕組みで動いているのかがわからないまま使ってると放置しがちですが・・・
**記事の管理とかアカウント管理周辺、おおかたこれでやってるんですよ。**
こいつが改ざんされたり変なもん仕込まれるリスクが高いんです。

ちなみにSQLってのは、例えば以下の構文で構成されるものです。

```
INSERT INTO wp_users
		(user_login, user_pass, user_email, user_url, user_registered)
	VALUES (stellorbit, Dfgris64454, shirube.factory@mkt27.com, https://stellorbit.net, 2026-03-29)

```

知識があまりないよ～って方のために説明すると、先頭に**「INSERT」**って書いてあると思います。つまり、データベース内に**「追加」**できるんです。
WordPressの構造は[ここらへんに詳しく載っていた](https://www.onamae.com/column/wordpress/45/)ので、普通に書きました。

ざっくりいうと、**侵入したいサイトにログイン情報を追加**して、侵入口を作れてしまうんです。

実際できるかどうかわかりませんし、 **やっちゃいけません。犯罪です。**

ちなみに侵入経路は、**問い合わせフォームとかコメント欄、検索欄とかのあらゆる入力欄**です。

この他にもメチャクチャな動きするマルウェア的なコード（**「悪意のあるコード」** とよく呼ばれるやつです）を突っ込んだりすることができたりします。

これを **「SQLインジェクション」** と言います。

あと、SQLではなく、Webサイト上や内部で悪人がコード書いてサイトを改ざんする **「XSS（クロスサイト・スクリプティング）」** という手法もあります。

#### 2.「遅い」「脆い」「手厚くない」は検索順位ダウン？

多分Webサイトを作ったことがない方でも **「SEO対策」** という言葉をよく聞くと思います。

手っ取り早く言うなら、**「検索サイトに引っ掛かりやすくする」** ためにアレコレやることです。

ただ、これも小手先の前にWebサイトの構造が駄目だとそもそもGoogleさんの機嫌を損ねたりします。

- `http://` ではなく `https://` にして、セキュリティ的に堅牢なものに設定する。（最近はブラウザが一旦止めてくるくらい重要）
- ページの表示はなるべく高速化すると有利
- 問い合わせフォームとかプライバシーポリシーを用意する

などなど・・・ちなみに上記のやつは、影響するとかしないとか **諸説あります。**

### 対策がぶっちゃけめんどいしやっても不安

これらの対策方法は主にこれらがあります。

- 中のエンジンの設定をする
- WordPressのテーマや中枢部にコード書き足す
- WordPressプラグインで対策をする

上2つが軽くていいんですが、正直面倒です。

かといって、あれもこれも・・・とWordPressプラグインでやると、**どんどん負荷が高くなって重くなります。**

### なにか良い策ない？とAIに聞いた

そこでnoteやはてなブログなんかに記事を上げていたのですが、**「月1000円ほどの課金しないと独自ドメインが使えない」「収益化が自由にできない」** など、かなりの制約がありました。

そんな中でぼんやり思っていたんです。

**「もうちょっと軽くて話が早くて、独自ドメイン使えて、収益化もできて、セキュリティ的にも問題なくて・・・Markdownとかで文章書けたらいいなー」** と、都合の良すぎる解決策を。

そこで解決策をAIチャットに聞いてみたら、「SSR」だの「SSG」だのと、カードのレアリティみたいな字面だけど、なかなかよさそうなものが解説されているサイトを発見しました。

2つ書いてありますが1個は割愛して、個人的に **「ブログサイト＋αで固定サイト」** という方式をとりたかったので、それに最適だといわれる選択肢をとりました。

それが **「静的サイトジェネレーター」、縮めてSSG** だったのです。

### SSGとは？

**「静的サイトジェネレーター（Static Site Generator）」** ってなんぞや？と今でも思っています。
調べてみると、 **「あらかじめ作ってあるやつを呼び出して高速化する」** 仕組みだそうです。

#### これまでのWebサイトとの違い

巷で溢れてるWebサイトの多くは、**アクセスするたびにページの構成要素をサーバーから読み込んでいる**ものが多いです。

建物とか製造に例えると、Webサイトを構成する資材を**いちいち運んできて、現場（ユーザーのPCやスマホなど）で組み立てている**ということです。

非常にオーソドックスな仕組みですが、この形式の問題点は、ろくに対策をしないと **「体感で遅く感じる」** ことです。

いくら回線が早くなっても、その分**コンテンツはリッチに、そして重く**なっていきます。

更に**提供元のサーバーの速度やユーザーに届くまでの経路に左右される**ため、どっかで貧弱なルートを辿ると、いくら**10Gbpsの光回線でもタルくなります。** 

それを解決するために、**CloudflareなどのCDN（コンテンツ配信ネットワーク、読んで字のごとし）を経由**したり、ユーザーがキャッシュを持つより先にサーバー上に**最初からキャッシュを形成**して高速化を図る等、様々な工夫が図られるのですが・・・

**正直めんどくさいです。** メインの記事を書くまでにたどり着く工数が多すぎて挫折することもしばしばありました。

#### SSGの利点

対してSSGでは、**読み込むものをあらかじめ決めて組み立ててあります。**

言わばこちらは、**「プレハブの建屋を丸ごと運んでくる」** とか、**「完成品をそのまま運んでくる」** 感じですね。

例えば車や家なんかを注文したとして、それが**整備済みの完成品としてそのまま家に届く**イメージです。

なんかのゲームで **「組みあがった住宅をヘリで丸ごと運ぶ」** みたいなのありましたが、そんな感じです。

実際に触れるとわかるんですが、記事やページが出来上がったら **「デプロイ」** という作業が必要になります。
デプロイのためのコマンドを実行すると、自分のPCやGitHubリポジトリ等で手早く組み立てて、それをアップロードしてしまいます。

外部から調達しなくても「完成した」ページを読んでいってるわけですから、**とにかく速い。**

この仕組みを使えば、**阿部寛のホームページ並みの速さのサイト** を誰でも作れるわけです。


### SSGのデメリットや向き不向き

良いことづくめのように見えますが、実はそうでもないんです。

#### SSGのデメリット①：速報性に弱い

メリットだらけかと思われますが、当然デメリットもあります。

この記事はAIツール「Perplexity」に聞き込みをしながら書いているのですが、その中でも個人的にデメリットに感じる部分がありました。

自分にはあまり関係のない項目ですが、**ニュースサイトや速報系のサイトと相性がメッチャ悪いです。**

例えば**YahooニュースみたいなサイトをSSGで作るとかかなり厳しい**です。

先述の通り、SSGは **「完成品を運んでくる」** タイプなのですが、完成品は**前もって組み立てなければいけません。**

つまり、例えば自分のこのサイトの場合だと、

大まかな工程として、 **「Markdownファイルを新規作成→日付データを入れる→本文を書く→ 組み立てる →公開」** という工程を経ています。

超高速で表示させるために前もって「組み立てる」という手順が必要で、これだけで十数秒はかかります。

この「組み立て」の工程のおかげで、前もってページを作っておいてそこから追記する野球の「一球速報」みたいなのでも**ちょっと遅くなります。**

つまり、**速報性に欠けるものは相性がよろしくないんです。**

どうしても速報性を重視したいなら、それなりの下準備が必要になると思います。

##### デメリットが関係ない人①

もちろん、このデメリット、関係ない人も大勢います。

例えば一例を挙げるなら、

- 小説やルポ・エッセイなど、**ある程度練り上げて書く文章**
- 自分のような、新製品などではない**速報性不要なレビュー記事**
- わざわざXとかインスタの**SNSに上げるほどでもないほどのお気持ち**

速報性の高いものを扱うのであれば、**WordPressなどのCMSを使用するか、SNSや動画・ライブ配信などの適した媒体はいくらでもあります。**

#### SSGのデメリット②：最初のセットアップが意外とめんどくさい

これが案外始めるのにハードル高いかもしれません。

自分は知識ゼロのままAIに聞きながらやったのですが、たまに間違ったこと教えられるせいでえらい目に遭いました。

**クラウドサービス（FirebaseやAzureなど）のログインや設定、**それに応じた**設定ファイルのチューニング** 等々…

それらが噛み合わないとエラー吐いてページが表示されないことがあります。

特に **「大文字・小文字の区別に厳しい」** という細かすぎてブチギレ必至の罠とか、発覚した時は頭抱えました。

なんとか乗り越えられればいいんですが、そこで折れちゃう人もいそうです。

次はデメリットではなく、個人的な反省点なんですが一応書いておきます。

## 個人的反省：Markdownが存外めんどいし自由度低い

正直、Markdownはあまり文章書くにはおすすめしたくないかな、というのが本音です。

画像を貼るにも一苦労、リンクの埋め込みもなんかできない（HTML通してるはずなのに・・・）、あと**ルールが多すぎる。**

### ルール一部紹介

実はDiscordでも使えるMarkdown記法なのですが、案外ルールが厳しいです。

それを一部紹介します。

#### 見出しは前後に空白行を入れる（MD022）

NG例

```
文章
### 見出し
文章
```

OK例

```
文章

### 見出し

文章
```

これ案外忘れがちです。

#### 見出しの末尾に句読点やピリオドはNG（MD026）

`##### これがダメな例です。`

#### HTML使用NG(MD033)

自分が使っている環境だとHTML使用可能になって、それで書いてある記事が一個だけありますが、基本ダメみたいです。

### 何が正解かいちいち調べてられない人へ

PC（Win/Mac/Linuxなんでもござれ）で無料で使用できる、**[「Visual Studio Code」](https://azure.microsoft.com/ja-jp/products/visual-studio-code)** に、
**「Markdown lint」** という拡張機能があります。

これはMarkdown記法のお作法を教えてくれるんですが・・・

自分は**警告音がうるさいので消しました。**

あとは「[Zettlr](https://www.zettlr.com/)」という、MarkdownのWYSIWIGエディタがあります。
こちらもちゃんと警告を出してくれるタイプ。
自分は**この記事を書き終わる直前にインストールして設定をしました。**

というわけで、いろいろと試行錯誤しながらこのサイトを更新している日々です。
